# Impeccable — Design Context

## Design Context

### Users
Neuroscientists, lab technicians, and students at research institutes (e.g. Gladstone / EthoML) who analyze animal-behavior video with the VAME machine-learning pipeline. **Many are not programmers** — the desktop app exists specifically so they can run VAME without the command line. Typical context: focused, multi-step analysis sessions in a research lab, working with large video and pose-estimation files and long-running ML jobs (training, segmentation) that can take many minutes. They value **correctness, reproducibility, and unambiguous status** far more than delight or novelty. The core job: create a project → preprocess → create trainset → train → evaluate → segment → community analysis → generate reports/visualizations, in sequence, exactly once each.

### Brand Personality
**Clinical & precise** — a serious scientific instrument, not a consumer app. Three words: **precise, trustworthy, unobtrusive**. Voice: factual, concise, neutral; never hype, marketing fluff, or emoji-driven cheer. Emotional goal: **confidence and control** — at every moment the user knows exactly what state the pipeline is in and what the next action is. The interface should recede so the data and the workflow are the focus.

### Aesthetic Direction
Refined **industrial / utilitarian-scientific**, executed with restraint. **Light theme only** — tinted cool neutrals, never pure white (#fff) or pure black (#000). Information-dense but calm and orderly. Data, status, and results are the heroes (logs, progress, evaluation images, motif/community videos), not decoration.

Reference feel: precision lab/pro tools and well-structured scientific dashboards — the restraint of tools like Linear, the clarity of good instrument UIs — but light and information-forward.

**Anti-references (explicitly avoid):**
- AI-slop: cyan-on-dark, purple→blue gradients, neon/glow accents, glassmorphism, gradient text, identical card grids, big-number "hero metric" layouts.
- Consumer-app playfulness, bounce/elastic motion, decorative icons above every heading.
- Monospace-everywhere as lazy "technical" shorthand (monospace belongs only in logs/paths/values).
- Wrapping everything in cards; nested cards.

### Proposed Palette (starting point — tinted cool neutrals + one purposeful accent)
- **Background**: `#F5F7FA` (cool light gray) · **Surface**: `#FCFDFE` · **Border**: `#E2E6EC`
- **Text primary**: `#1A2230` · **Text secondary**: `#5A6473`
- **Accent / primary action & interactive**: deep petrol teal `#0F6E78` (hover `#0B5159`) — distinctive and scientific, deliberately *not* the default AI cyan/blue.
- **Status (reserved strictly for state — never decoration):** success `#1F8A4C` (green) · failed `#C2392E` (red) · running/attention `#B7791F` (amber).
- Tint neutrals toward the accent hue for subconscious cohesion; status colors only ever signal pipeline state.

### Design Principles
1. **Status is sacred.** Every long-running step shows clear, honest state at a glance — running / success / failed — never ambiguous, never a green check when something errored. Status colors (green/red/amber) are reserved exclusively for this.
2. **Sequence with clarity.** The linear 1→6 pipeline is the backbone. The UI always makes the *current* step and the *next* action obvious; reveal complexity progressively (accordions, advanced options behind disclosure), never dump it all at once.
3. **Tinted neutrals + one accent.** A cohesive light palette anchored in cool-tinted grays with a single functional accent (petrol teal) reserved for primary actions and in-progress; no gratuitous color.
4. **Density without noise.** Dense by necessity (forms, parameters, logs) but ordered through consistent spacing rhythm, real type hierarchy, and minimal containers — flatten, don't nest.
5. **Calm, factual voice.** Concise, neutral microcopy. Errors are specific, actionable, and non-blaming. No hype, no emoji.

### Accessibility
Target **WCAG 2.1 AA** contrast for text and UI. Status must never rely on color alone — pair with an icon/label (✓ / ✗ / spinner + text). Respect `prefers-reduced-motion`. Ensure keyboard focus is visible and the file-browser/forms are keyboard-navigable.
